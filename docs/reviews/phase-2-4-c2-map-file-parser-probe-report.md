# Phase 2.4-C2 Map File Parser Probe Report

## Overview
This report documents the results of the isolated Parser Probe (Phase 2.4-C2). The goal was to build a robust, isolated parser capable of reading a `.map` file locally generated by the map generation probe.

## Input File
- **File:** `D:\RustMasterToolMapGen\output\proceduralmap.4750.1321.286.map`
- **Size:** 47,100,648 bytes (44.92 MB)
- **SHA256:** `C7AB7FF1D6C599D5B5D20F1D1D33EFED7C6932DE5E05946DF38DAB4E5DC3CFD0`

## Parser Approach
- **Language/Environment:** Node.js (JavaScript)
- **Libraries Used:** Standard `fs`, `crypto`, and `lz4js`
- **Location:** `experiments/map-parser-probe/`
- **Strategy:** Read file size, compute hash, extract a hexdump of the header, attempt standard LZ4 decompression, and perform a raw binary scan for known Protobuf/Rust map string identifiers.

## Reconnaissance & Header Analysis
The first 256 bytes were dumped and analyzed:
```text
00000000  0a 00 00 00 78 4f 7f 48 9f 01 00 00 03 80 80 40   |....xO.H.......@|
00000010  e5 ae 03 ff 0a 08 8e 25 12 90 80 81 90 00 0a 07   |e......%........|
00000020  74 65 72 72 61 69 6e 12 82 80 81 10 99 39 02 00   |terrain......9..|
```
- The string `terrain` is clearly visible at offset 32, preceded by `0A 07` (which corresponds to Protobuf Field 1, wire type 2, length 7).
- This indicates that the file structure heavily aligns with the expected `WorldSerialization` Protobuf format.

## Decode / Decompression Attempts
Rust map files are known to use LZ4 compression, often wrapped by C#'s `lz4net`.
- **Attempt 1:** Standard LZ4 frame decompression via `lz4js`.
- **Result:** FAILED (`invalid magic number`).
- **Conclusion:** The file does not use standard LZ4 framing (`0x184D2204`). It likely uses a raw LZ4 block format (often indicated by a leading 4-byte uncompressed size header) or a custom Rust specific wrapper that standard JS LZ4 decoders reject.

## Structure Extraction Results
Despite the decompression failure, a raw byte scan of the 45MB file successfully extracted plaintext identifiers matching Rust's map layers:
- `terrain` (offset 32)
- `height` (offset 18,608,460)
- `splat` (offset 37,289,620)
- `biome` (offset 43,366,756)
- `topology` (offset 45,204,599)
- `alpha` (offset 46,805,017)
- `water` (offset 46,826,582)

**Mentions found:**
- `path`: 2 times
- `prefab`: 0 times
- `monument`: 0 times

*Note:* The absence of the string `prefab` in plaintext implies that prefab records are either stored under a different Protobuf structure, compressed successfully within LZ4 blocks without string literals leaking, or mapped via integer String IDs.

## Assessment
- **Terrain/Height/Topology/Water:** Realistically reachable. The layers exist sequentially in the file. With a dedicated C#/Rust LZ4 block parser, the raw byte arrays can be extracted.
- **Prefabs/Monuments:** Realistically reachable, but will require full protobuf schema decoding and ID mapping (StringPool).
- **Paths/Roads/Rivers:** Realistically reachable through the same Protobuf decoding.
- **Resources (Stone/Sulfur/Metal):** These are usually procedurally generated at runtime based on the `splat` and `biome` maps. They cannot be directly extracted as prefab entities, but we can generate accurate density models by reading the `splat` layer.

## Final Verdict & Success Level
**Success Level:** **Level B — Strong Success**
We successfully read the file, identified the Protobuf structure, and extracted all major terrain layer names from the binary. The lack of a compatible JS LZ4-block decoder blocked Level C, but the data presence is proven.

**Verdict:** **GO FOR C3 PARSER HARDENING**
The next smallest real implementation step is to migrate the parser to a robust Rust (`lz4_flex` + `prost`) or C# environment where raw LZ4 block decompression and Protobuf schema definitions (`Rust.WorldSerialization`) can be natively applied.
